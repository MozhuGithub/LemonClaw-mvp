### 3. 架构方案决策（核心工作）

系统对比三种集成 OpenClaw 的方式：

| 方案 | 初始成本 | 维护成本 | 升级难度 | 性能 |
|------|---------|---------|---------|------|
| Bundle 模式（HomiClaw） | 极高 | 高 | 低 | 最快 |
| **Vendor 子进程（RivonClaw）** | **低** | **低** | **中** | **快** |
| 渐进式 | 低 | 极高（推倒重来） | - | 渐变 |

**Bundle 模式为什么成本极高**（深入分析 OpenClaw 构建系统）：
- tsdown 构建 + 20 个动态 import 入口（createLazyRuntimeModule）
- jiti 运行时编译器（.ts 文件运行时编译）
- 90+ 扩展的动态发现机制
- 多个 native 模块（sharp、@napi-rs/canvas）
- pnpm workspace monorepo 结构

**最终决策：Vendor 子进程模式**
- 低初始成本：npm install + 子进程启动
- 进程隔离：Gateway 崩溃不影响 Electron UI
- 升级简单：pin 版本 + adapter 适配
- 开发调试快：Gateway 独立重启
